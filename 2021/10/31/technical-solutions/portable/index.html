Portable OJ 一期工程技术方案

架构设计

系统结构

采用三端分离的架构

web

  • 网页前端，通过 HTTP/HTTPS 协议与 server 进行通信

server

  • 负责与前端进行 HTTP/HTTPS 协议通信
  • 负责逻辑任务与定时任务的处理
  • 与 judge 通过 TCP 长链接与非阻塞 IO 进行通信
  • 负责文件的管理和数据库的连接

judge

  • 负责进行程序的编译和运行

存储结构

所有模块的需要索引的信息和部分短字段均存储至 MySQL，非短字段或者非需要索引字段均存储至 Mongo，Redis 仅用于缓存且不采用持久化

模块设计

用户模块

用户属性

  • id。唯一不变的用户标识
  • 用户名。表示用户的唯一标识，仅限大小写英文、数字和下划线组成，长度至少三个字符，通常情况下不允许更新
  • 密码。仅限大小写英文、数字和!@#$%^&*()_+-=[]\;',./{}|:"<>?组成
  • 类型。默认为 NORMAL
  • 组织。用户一定属于某个组织，组织按层级高层级对低层级有管理权，对同层级没有管理权，管理关系以 DAG 图表示。能够进行管理的组织可以调整对方的所在组织和所拥有的权限
  • 权限。用户拥有的权限，不同的权限组成的数组
  • 总提交数。
  • 总通过数。
  • 邮箱。用于找回密码和后续的通知信息，在第一次版本迭代中不发挥作用

用户功能

  • 通过 handle 和 password 登录
  • 通过 handle 和 password 注册
  • 调整对方所属组织
    • 对方所属组织以及被调整后的组织必须要在都在当前用户的管理下
  • 授予权限（需要 GRANT 权限）
    • 被授予的权限必须是当前用户拥有的权限，且对方在当前用户的管理下
  • 收回权限（需要 GRANT 权限）
    • 被收回的权限必须是当前用户拥有的权限，且对方在当前用户的管理下
